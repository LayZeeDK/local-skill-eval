# Phase 4.1: Tune Ollama Agent to 5 Min Trial Average - Research

**Researched:** 2026-03-10
**Domain:** Ollama parameter tuning, prompt engineering for small-model tool agents, benchmarking methodology
**Confidence:** HIGH

## Summary

This phase aims to reduce the superlint_demo average trial time from ~440-590s down to 300s (5 minutes) while maintaining a reward of 0.90+. The optimization surface spans four domains: (1) Ollama server and Modelfile parameters, (2) system prompt engineering, (3) agent configuration (maxIterations, maxToolOutputChars), and (4) alternative model exploration. Current baseline uses qwen3:4b with temperature 0, num_ctx 4096, num_predict 4096, and a 3-line system prompt.

Key findings: temperature 0 is suboptimal for agent workloads (should be 0.2-0.4 for tool-calling precision with slight flexibility); the current system prompt is too short and lacks structure that helps small models follow tool-calling patterns; context pruning of old tool results is a high-impact optimization for reducing per-turn prompt eval time; and Qwen 3.5:4b (released March 2, 2026) is a promising fallback with significantly improved instruction following and tool-calling accuracy over Qwen 3:4b.

**Primary recommendation:** Build a benchmark script that captures per-turn Ollama metrics, then systematically test parameter combinations starting with (1) temperature 0.25, (2) reduced num_predict with /no_think, (3) structured system prompt, and (4) context pruning -- before exploring alternative models.

### Pattern 1: Benchmark Script That Wraps Eval
**What:** A standalone script that runs `npm run eval -- superlint_demo --agent=ollama --provider=local --trials=3` with different Modelfile/env configurations, capturing wall-clock time and Ollama metrics.
**When to use:** Every benchmark experiment.
**How it captures metrics:** The Ollama API returns per-request metrics in the ChatResponse:
```typescript
// Ollama ChatResponse includes these timing fields:
interface ChatResponse {
  // ... message fields ...
  total_duration: number;      // nanoseconds
  load_duration: number;       // nanoseconds - time to load model
  prompt_eval_count: number;   // tokens evaluated in the prompt
  prompt_eval_duration: number; // nanoseconds for prompt evaluation
  eval_count: number;          // tokens generated
  eval_duration: number;       // nanoseconds for generation
}
```
**Approach:** Since we cannot modify OllamaToolAgent, the benchmark script should either:
1. Parse the eval CLI output (wall-clock time and command count are already printed)
2. Create a lightweight wrapper that intercepts Ollama API responses via a proxy or by extending the agent for benchmark mode only

**Recommendation:** Option 1 (parse CLI output) for wall-clock + reward, plus a separate direct-API profiling script that runs a single trial with instrumented Ollama calls to capture per-turn metrics.

### Pattern 2: Experiment Matrix
**What:** Systematically vary one parameter at a time against the baseline, recording results.
**When to use:** When exploring the parameter space.
**Example matrix:**
```toml
[baseline]

temperature = 0
num_ctx = 4096
num_predict = 4096
system_prompt = "current"

[experiment.temp_025]

```

### Pattern 3: Modelfile Swapping for Experiments
**What:** Create multiple Modelfiles for different configurations, use `ollama create` to swap between them.
**When to use:** For each parameter experiment that requires Modelfile changes.
**Example:**
```bash
# Create variant for testing
ollama create qwen3-skill-eval-agent -f modelfiles/qwen3-skill-eval-agent.experiment.Modelfile
# Run benchmark
npm run eval -- superlint_demo --agent=ollama --provider=local --trials=3
# Restore baseline
ollama create qwen3-skill-eval-agent -f modelfiles/qwen3-skill-eval-agent.Modelfile
```

### Anti-Patterns to Avoid
- **Changing multiple parameters at once:** Impossible to attribute improvements to specific changes. Change one variable per experiment.
- **Skipping warm-up run:** Cold-start model loading adds 10-30s noise. Always do one warm-up trial before measured trials.
- **Using OLLAMA_FLASH_ATTENTION blindly:** Phase 2 research documented that these env vars helped in CI but had no effect or slowed inference on the local Snapdragon X Elite. Must benchmark, not assume.
- **Putting system prompt in Modelfile SYSTEM directive:** Known bugs where API system message doesn't reliably override Modelfile SYSTEM. Keep system prompt in API calls (current approach) for flexibility.

## Common Pitfalls

### Pitfall 1: Prompt Eval Time Grows Quadratically with Conversation Length
**What goes wrong:** Each turn re-evaluates the full conversation history. By turn 9, the prompt includes all prior tool calls and results, making prompt_eval_duration the dominant cost.
**Why it happens:** Ollama/llama.cpp re-processes the full prompt on each call (with KV cache reuse for common prefixes, but tool results change the suffix).
**How to avoid:** Context pruning: truncate or summarize old tool results after N turns. Keep system prompt + last 2-3 exchanges intact.
**Warning signs:** prompt_eval_duration growing significantly per turn; later turns taking 2-3x longer than early turns.

### Pitfall 2: Temperature 0 Causes Deterministic Loops
**What goes wrong:** With temperature 0, the model picks the same greedy path every time. If a tool call fails or returns unexpected output, the model repeats the exact same call.
**Why it happens:** No sampling randomness to escape local optima. Phase 4 found that an aggressive system prompt caused 18-command loops.
**How to avoid:** Use temperature 0.2-0.4 for agent workloads. This provides enough randomness to escape loops while maintaining tool-calling precision.
**Warning signs:** Agent repeating the same command 3+ times; hitting maxIterations without completing.

### Pitfall 3: Modelfile SYSTEM Directive Conflicts with API System Message
**What goes wrong:** When both Modelfile SYSTEM and API system message are provided, behavior is inconsistent. Some Ollama versions ignore the API system message and use the Modelfile one.
**Why it happens:** Known Ollama bug (issues #2470, #8729). The /api/chat endpoint doesn't reliably override Modelfile SYSTEM.
**How to avoid:** Do NOT use Modelfile SYSTEM directive for the agent. Keep system prompt exclusively in the API `messages` array (current approach is correct). Use Modelfile only for parameters (num_ctx, num_predict, temperature, num_thread).
**Warning signs:** System prompt changes having no effect on model behavior.

### Pitfall 4: num_predict Too Low Truncates Tool Calls
**What goes wrong:** Tool call JSON is cut off mid-generation, producing invalid JSON that Ollama can't parse as a tool_call.
**Why it happens:** num_predict limits total output tokens per response. Complex tool calls with long arguments (like write_file with content) can easily exceed 1024 tokens.
**How to avoid:** Keep num_predict at 2048+ for agent workloads. Test with the most token-heavy tool call pattern (write_file with multi-line content).
**Warning signs:** Agent loop receiving empty tool_calls; model generating partial JSON in content field.

### Pitfall 5: Benchmark Variance from System Load
**What goes wrong:** Trial results vary wildly (>30% variance) making it impossible to compare configurations.
**Why it happens:** Other processes (Windows Update, background services, Ollama model reloading) consume CPU during trials.
**How to avoid:** Close non-essential apps, use warm-up runs, take 3 trials minimum, report standard deviation, run all experiments back-to-back in one session.
**Warning signs:** Standard deviation > 20% of mean across trials.

### Pitfall 6: Qwen 3.5 Tool Calling Requires Ollama v0.17.7+
**What goes wrong:** Qwen 3.5 tool calls are parsed incorrectly, model prints tool call JSON as text instead of structured tool_calls.
**Why it happens:** Earlier Ollama versions routed Qwen 3.5 through the wrong parsing pipeline (Hermes-style JSON instead of Qwen3-Coder XML format).
**How to avoid:** Ensure Ollama is v0.17.7 or later before testing Qwen 3.5 models.
**Warning signs:** Smoke test fails; model output contains `<tool_call>` XML tags as plain text.

## Ollama Parameter Research

### Temperature (MEDIUM-HIGH confidence)
| Value | Use Case | Source |
|-------|----------|--------|
| 0.0 | Deterministic grading, consistent scoring | Phase 4 Modelfile (grader-oriented) |
| 0.2-0.25 | Precise tool-calling with slight flexibility | Community Modelfile for qwen3 agent use (mychen76/qwen3_cline_roocode) |
| 0.4 | More explorative agent behavior | General agent guidance |
| 0.7 | Creative tasks, general chat | Qwen official recommendation for general use |

**Recommendation for experiment matrix:** Test 0.0, 0.25, 0.4. Start with 0.25 as primary candidate.

### num_predict (HIGH confidence)
Current: 4096. This is generous. With `/no_think` suppressing thinking tokens, the model needs far fewer output tokens per turn. A typical tool call is 50-200 tokens.

**Recommendation:** Test 1024, 2048, 4096. Lower values mean Ollama generates fewer tokens before stopping, which directly reduces generation time. If `/no_think` is reliably suppressing thinking, 2048 should be safe. 1024 is aggressive but worth testing for simple tool calls.

### num_ctx (HIGH confidence)
Current: 4096. The superlint_demo task runs ~9 turns. With 4 tools and their schema (about 800 tokens), system prompt (~80 tokens), instruction (~200 tokens), and accumulated tool results, context fills up fast.

**Recommendation:** Test 4096, 8192. Higher context allows more history but increases prompt_eval_duration. If context pruning is implemented, 4096 may suffice. Without pruning, 8192 gives headroom.

### num_batch (MEDIUM confidence)
Controls how many tokens are processed in parallel during prompt evaluation. Higher values use more memory but speed up prompt eval. Default is 512 in llama.cpp.

**Recommendation:** Test 256, 512, 1024. On Snapdragon X Elite with 32GB RAM, 1024 should be safe and may speed up prompt eval for large contexts.

### repeat_penalty (MEDIUM confidence)
Penalizes repeated token sequences. Default is 1.1 in Ollama. For agent workloads, mild repeat penalty (1.05-1.1) can help prevent command loops without hurting legitimate repetition in code.

**Recommendation:** Test 1.0 (off), 1.05, 1.1. If temperature 0.25 already prevents loops, repeat_penalty may be unnecessary overhead.

### top_k and top_p (MEDIUM confidence)
Community recommendations for Qwen3 agent use: top_k=40, top_p=0.9. Qwen official for Qwen3-Coder: top_k=20, top_p=0.8.

**Recommendation:** Include in matrix only if temperature changes alone don't hit target. Lower priority than temperature and num_predict.

### mirostat (LOW confidence)
Adaptive sampling algorithm. May help with consistency but adds compute overhead. Not commonly used for tool-calling agents.

**Recommendation:** Skip unless other parameters fail to hit target.

### OLLAMA_FLASH_ATTENTION and OLLAMA_KV_CACHE_TYPE (HIGH confidence for experiment inclusion)
Phase 2 research: helped CI runners (12s to 6.3s for grading) but had NO effect on local Snapdragon X Elite. User reports flash attention SLOWED local inference.

**Recommendation:** Include in experiment matrix with strict A/B comparison. Test: (1) neither set, (2) both set (OLLAMA_FLASH_ATTENTION=1 + OLLAMA_KV_CACHE_TYPE=q8_0). Do NOT apply blindly.

### OLLAMA_NUM_PARALLEL (HIGH confidence)
Default: 1. For serial agent workloads, 1 is correct. Higher values pre-allocate context for unused parallel slots, wasting RAM and potentially slowing inference.

**Recommendation:** Keep at 1. Do not experiment.

### stream: true vs false (HIGH confidence)
Current: `stream: false`. Research confirms:
- Wall-clock generation time is the same regardless of streaming mode
- Ollama's native API (`/api/chat`) now supports streaming + tool calling (May 2025+)
- The `ollama` npm package handles streaming correctly
- No performance benefit to switching

**Recommendation:** Keep `stream: false`. No experiment needed.

### keep_alive Strategy (HIGH confidence)
For benchmarking: use `keep_alive: -1` (keep model loaded indefinitely) during the benchmark session to eliminate reload latency between trials. Use a warm-up run to pre-load the model.

For production: current `keep_alive: 0` (unload immediately after run) is correct because the grader model needs to load next.

**Recommendation:** Benchmark script should set `keep_alive: -1` at the start of the session, then `keep_alive: 0` on the final trial to unload. Current agent code already uses `keep_alive: 0` in the finally block, which is correct for production.

## System Prompt Engineering

### Current Prompt
```
You are an AI agent that completes coding tasks. Use the provided tools to complete the task.
Do not explain your reasoning - just call the appropriate tool.
When you are done, respond with a summary of what you did. /no_think
```

### Analysis
- Too generic -- gives no structure for how to approach tasks
- "Do not explain your reasoning" is good for reducing token waste
- Missing: step-by-step guidance that helps small models plan
- Missing: explicit instruction to not repeat failed commands
- `/no_think` is correct for thinking suppression

### Recommended Prompt Structure (based on eval article research)
```
You are a coding agent. Complete the task using the provided tools.

Rules:
- Call tools directly, no explanations between calls.
- Read files before editing them.
- If a command fails, try a different approach instead of repeating it.
- When finished, respond with a brief summary.

/no_think
```

**Rationale:** Anthropic's eval guide emphasizes that structured, concise prompts with clear rules outperform vague instructions for agent tasks. OpenAI's eval skills guide notes that prompt engineering for agent skills should be tight and prescriptive for small models. The key additions are: read-before-edit (reduces wasted turns) and don't-repeat-failures (prevents loops).

### Few-Shot Examples: NOT Recommended (MEDIUM confidence)
Research found no strong evidence that few-shot examples reduce turns for small-model tool agents. The token cost is significant (each example adds 200-500 tokens to every turn's prompt eval) and the benefit is uncertain. For a 4B model with 4096 context, few-shot examples would consume 5-12% of the context budget.

**Recommendation:** Do not add few-shot examples. Focus on clear, structured zero-shot instructions.

## Context Pruning Strategies

### Why Context Pruning Matters
The superlint_demo task runs ~9 turns. By turn 7-8, the conversation history contains all prior tool calls and their full results. Each turn re-evaluates the full prompt, and prompt_eval_duration grows linearly with context size.

### Strategy 1: Observation Masking (Recommended)
Replace old tool result content with a short summary after N turns.

**Implementation:**
```typescript
// Before sending to Ollama, prune messages older than keepLastN turns
function pruneHistory(messages: Message[], keepLastN: number = 4): Message[] {
  const system = messages[0]; // Always keep system prompt
  const user = messages[1];   // Always keep original instruction
  const recent = messages.slice(-keepLastN * 3); // 3 messages per turn: assistant + tool_calls + tool results

  if (messages.length <= keepLastN * 3 + 2) {
    return messages; // No pruning needed
  }

  // Replace middle messages with summary
  const pruned = messages.slice(2, -keepLastN * 3);
  const summary = `[Previous ${pruned.length} messages pruned. The agent has been executing tool calls to complete the task.]`;

  return [system, user, { role: 'user', content: summary }, ...recent];
}
```

**JetBrains research confirms:** Observation masking matched LLM-based summarization in both cost savings and problem-solving ability for coding agents.

### Strategy 2: Tool Output Truncation (Already Implemented)
Current `maxToolOutputChars: 8000` is already truncating. Consider whether 4000-6000 would suffice for superlint_demo's outputs (lint results, file contents).

### Strategy 3: Compaction (Complex, defer)
Use a separate LLM call to summarize conversation history. Too complex for Phase 4.1 -- this is an escalation path.

**Recommendation:** Implement Strategy 1 (observation masking) with keepLastN=4. If insufficient, reduce maxToolOutputChars. Compaction is out of scope.

**NOTE:** Context pruning requires modifying OllamaToolAgent internals. CONTEXT.md says "do not modify OllamaToolAgent internals for metric capture" but context pruning is an optimization, not metric capture. If the planner interprets this strictly, context pruning becomes an escalation path rather than a first-pass optimization.

Current approach: dual suppression (`think: false` in API + `/no_think` in system prompt). Alternative: use a non-thinking model like qwen2.5:7b which has no thinking mode at all.

Non-thinking models (qwen2.5:7b) avoid the issue entirely but are larger (4.7GB vs 2.5GB for qwen3:4b). Worth testing as a fallback model.

- Test file uses: 10

The task has 3 mandatory steps (check, fix, verify) plus file reads and directory listing. A well-performing agent should complete in 6-10 turns. Setting maxIterations to 15-20 provides headroom for retries without allowing runaway loops.

## NPU Investigation

### Finding: No Practical Ollama NPU Path (HIGH confidence)

The Snapdragon X Elite NPU (45 TOPS) cannot be used with Ollama:

1. **Ollama uses GGUF format; NPU requires ONNX.** The Qualcomm Hexagon NPU is designed for ONNX models via Qualcomm AI Engine Direct. GGUF models (used by Ollama/llama.cpp) are not compatible.

2. **No GPU backend either.** Ollama on Windows ARM64 is CPU-only. There is no DirectML GPU backend for ARM64 in llama.cpp.

3. **Alternative paths exist but are out of scope:** AnythingLLM + ONNX models or VS Code AI Toolkit can use the NPU, but these are entirely different inference stacks, not Ollama.

4. **Snapdragon X2 (2026)** doubles NPU to 80 TOPS but software ecosystem maturity is the bottleneck, not hardware.

**Recommendation:** Defer NPU acceleration entirely. No easy win exists for Ollama on Snapdragon X Elite.

## Modelfile SYSTEM vs API System Message

### Finding: Use API System Message Only (HIGH confidence)

Research confirms the current approach is correct:

1. **Modelfile SYSTEM** bakes a system prompt into the model at creation time.
2. **API system message** (role: 'system' in messages array) is supposed to override it, but there are known bugs (Ollama issues #2470, #8729) where the override doesn't work reliably with `/api/chat`.
3. The current codebase uses API system message exclusively (SYSTEM_PROMPT constant sent in messages array), with NO Modelfile SYSTEM directive. This is the correct approach.
4. **Tools cannot be defined in Modelfile** -- they must be passed via API. This further reinforces keeping everything in API calls.

**Recommendation:** Do not add SYSTEM to the Modelfile. Continue using API system message. Keep Modelfile for parameters only (num_ctx, num_predict, temperature, num_thread).

### Per-Turn Metric Profiling Script
```typescript
// scripts/profile-agent-turns.ts
import { Ollama } from 'ollama';

// Runs a single agent interaction with instrumented per-turn metrics
// Does NOT replace OllamaToolAgent -- separate profiling tool
const client = new Ollama({ host: 'http://localhost:11434' });

async function profileTurn(
  messages: any[],
  tools: any[]
): Promise<{
  eval_count: number;
  eval_duration: number;
  prompt_eval_count: number;
  prompt_eval_duration: number;
  total_duration: number;
}> {
  const response = await client.chat({
    model: 'qwen3-skill-eval-agent',
    messages,
    tools,
    stream: false,
    think: false,
  });

  return {
    eval_count: response.eval_count ?? 0,
    eval_duration: response.eval_duration ?? 0,
    prompt_eval_count: response.prompt_eval_count ?? 0,
    prompt_eval_duration: response.prompt_eval_duration ?? 0,
    total_duration: response.total_duration ?? 0,
  };
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Temperature 0 for all Ollama models | Temperature 0.2-0.4 for agent workloads | Community practice 2025 | Prevents deterministic loops while maintaining precision |
| Full conversation history every turn | Context pruning / observation masking | JetBrains research Dec 2025 | Matches summarization quality; reduces prompt eval time |
| Qwen3:4b best small agent model | Qwen3.5:4b with better instruction following | March 2, 2026 release | Improved tool calling, instruction following benchmarks |
| Ollama tool calling streaming broken | Native API streaming with tools works | Ollama May 2025+ | No impact for us (we use stream: false) |

**Deprecated/outdated:**
- Temperature 0 for agent tasks: creates deterministic loops; use 0.2-0.4
- Qwen3 as newest small model: Qwen3.5 supersedes it with better agentic capabilities
- Relying on `/no_think` alone: use `think: false` API parameter (already implemented)
